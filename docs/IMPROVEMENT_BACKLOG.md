## 效能與載入

- **寵物圖檔**：`public/pets/` 合計約 **30MB**，多張 PNG 單檔 **0.5～1.7MB**；畫面只顯示小圖仍會整張下載。建議用 Squoosh／oxipng 等壓縮、必要時降解析度，或評估 **WebP** + 建置流程。
- **字型**：`index.html` 自 Google Fonts 載入多個字重；若極在意首屏，可改 **子集字型自託管** 或縮減字重。

---

## 部署與 CI

- **前後端同步部署**：目前 GitHub Actions 只部署 **GitHub Pages**；**Render** 另靠儲庫連動自動部署。若要「一次 push 保證兩邊版本一致」，可評估在 workflow 裡呼叫 Render Deploy Hook，或文件化必推兩邊的流程檢查清單。

## 對戰與養成設計（較大功能）

- **物種差異**：戰鬥結算尚未讀 `species`；可擴充為物種係數、專屬招或第四招，且需伺服器白名單驗證。
- **MP／與養成連動的 HP**：戰鬥已有 HP；若要 MP 或養成數值影響戰鬥，需在 `Battle` 狀態與 `resolveRound` 擴充並更新 UI／規則文件。

---

## 小項／技術債（可選）

- **`phase` 狀態**：部分畫面切換時 `phase` 未必與實際畫面同步（例如回養成）；若之後依 `phase` 做邏輯，建議統一在 `renderCare`／`renderLobby` 等入口更新。
- **照顧冷卻**：目前**所有**照顧共用約 1.1 秒間隔；若產品希望「只有休息較長、餵食較短」，可拆成不同 `GAP_MS`。
- **`list_open_rooms` 防濫用**：若擔心刷請求，可加簡單節流（每連線每秒最多 N 次）。
